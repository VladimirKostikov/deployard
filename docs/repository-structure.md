# Repository Structure

This repository follows the layout used by mature open-source monorepos (Turborepo, Nx, CNCF projects).

## Top-level layout

```
.
├── apps/                   Deployable applications
│   ├── api/                NestJS backend
│   └── web/                React frontend
├── packages/               Shared libraries
│   └── shared/             TypeScript types and contracts
├── deploy/                 Deployment artifacts
│   ├── docker/             Dockerfiles, nginx, compose
│   └── helm/dpd/           Helm chart (planned)
├── docs/                   Project documentation
├── examples/               Runnable samples
│   └── kubernetes/         kind config, demo manifests
├── scripts/                Ops scripts (prod-relevant)
│   ├── docker/             Container entrypoints
│   └── deploy/             Image rollout helpers
├── tests/                  Cross-package tests
│   ├── e2e/                End-to-end tests
│   └── integration/        API integration tests
├── .github/                GitHub templates and CI
├── Makefile                Developer commands
├── package.json            Workspace root
└── pnpm-workspace.yaml     Monorepo definition
```

## What lives where

| Directory | Purpose | Deployable |
|---|---|---|
| `apps/` | Runnable products | Yes |
| `packages/` | Code shared between apps | No |
| `deploy/` | How to ship the product | No |
| `docs/` | Architecture, ADRs, guides | No |
| `examples/` | Copy-paste samples for users | No |
| `scripts/` | Docker entrypoints and deploy helpers | No |
| `tests/` | Quality gates across apps | No |

## Why `apps/` not `src/`

International convention for JavaScript/TypeScript monorepos:

- `apps/` — things you deploy (API, web, CLI)
- `packages/` — things you import (types, UI kit, config)

Used by Vercel Turborepo, Nx, Google, and most top GitHub monorepos.

Application source code stays inside each app (`apps/api/src`, `apps/web/src`). The root directory stays thin and documentary.

## Root directory policy

Allowed at repository root:

- README, LICENSE, CHANGELOG, SECURITY
- Workspace config (`package.json`, `pnpm-workspace.yaml`, `pnpm-lock.yaml`, `tsconfig.base.json`)
- Tooling (`.prettierrc`, `Makefile`, `.env`)
- `.github/`, `.cursor/`

Not allowed at root:

- Application source code
- Random config dumps
- One-off scripts (use `scripts/`)

Not committed to git:

- `node_modules/` (generated by pnpm)
- `dist/`, `build/`, `coverage/`

## Root files explained

| File / folder | Stays at root? | Why |
|---|---|---|
| `package.json` | Yes | Workspace orchestrator: `pnpm dev`, `pnpm build` for all packages |
| `pnpm-workspace.yaml` | Yes | pnpm requires this at repo root to find `apps/*` and `packages/*` |
| `pnpm-lock.yaml` | Yes | Single lockfile for the entire monorepo — reproducible installs |
| `node_modules/` | Yes (generated) | pnpm hoists dependencies to root; gitignored |
| `Makefile` | Yes | Entry point for docker, kind, cluster — like in Kubernetes, Terraform repos |
| `tsconfig.base.json` | Yes | Shared TypeScript baseline extended by each app |
| `.prettierrc` | Yes | One formatter config for the whole repo |
| `.env` | Yes | Local dev defaults for Docker Compose |
| `apps/*/package.json` | Inside each app | Dependencies specific to api or web |
| `apps/*/node_modules/` | Inside each app (optional) | pnpm may symlink; usually root `node_modules` is enough |

`package.json` at root is **not** application code. It only delegates:

```json
"dev": "pnpm --parallel --filter @dpd/api --filter @dpd/web run dev"
```

Real dependencies (`@nestjs/common`, `react`, etc.) live in `apps/api/package.json` and `apps/web/package.json`.
